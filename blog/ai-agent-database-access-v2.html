I Gave an AI Agent My Database Password

April 2026 · 8 min read

TL;DR

I built an inline proxy between AI agents and Postgres that checks every SQL query before it hits the database. Then I set up an AI-vs-AI red team — one agent tries to steal data, the proxy tries to stop it, hundreds of rounds, no human in the loop. First run: the attacker got through 674 times. A week of fixes later: zero. The tool is FaultWall — one Go binary, one YAML config, MIT licensed.

The problem

I had a LangChain agent talking to Postgres 16. Claude as the LLM, SQLDatabaseToolkit, customer data in the database. It wrote SQL, pulled feedback summaries, joined product tables. Worked fine for three weeks.

Then it generated SELECT * FROM feedback with no WHERE clause. Two million rows. CPU pinned for three minutes. The connection came from the same agent_user role that runs normal queries all day, so there was nothing in the metrics to distinguish "LLM went sideways" from "application doing its job."

That was the harmless version. I started poking around with the agent's own credentials:

  • DROP TABLE users; worked. (Rolled it back.)
  • SELECT * FROM customers; returned names, emails, phone numbers. The agent only ever needed the feedback table.
  • SELECT pg_read_file('/etc/passwd'); read server files through Postgres.
  • SELECT * FROM dblink('host=...', 'SELECT 1') made an outbound connection to an external database.

All valid SQL, all executed with the agent's credentials. This is the confused deputy problem: the agent is authorized, the query it generates might not be, and Postgres doesn't know the difference.

Postgres roles are static. My agent has the same permissions summarizing feedback as it does deleting records. Row level security helps, but you're trusting the model to include the right predicates. Neither stops pg_sleep(999) or lo_export() or dblink().

The sidecar that didn't work

My first attempt was a sidecar. Connect to Postgres as a monitoring user, poll pg_stat_activity every 10 seconds, kill violating queries with pg_terminate_backend().

I thought this was fine. Agent queries tend to run for 5-30 seconds. A 10 second poll should catch most of them.

A friend who's a CTO at a small startup pointed it at their staging database and ran five queries that touched PII. FaultWall caught one. The other four finished between polling intervals. The one it caught was a pg_sleep(15) that was deliberately slow.

One out of five. That's not a firewall, that's a coin flip with bad odds. Polling works for audit logs and dashboards. For enforcement, I needed to be in the query path.

The proxy

FaultWall is now an inline L7 proxy. It speaks the Postgres wire protocol via pgproto3, sits between your agents and the database, and parses every query before forwarding it. If the query violates policy, FaultWall returns an error. The query never reaches Postgres.

Agents connect to FaultWall on port 5433. FaultWall connects to Postgres on 5432. Agents never get the real database credentials. Every query goes through pg_query_go, which wraps Postgres's own C parser. It extracts the operation type, every table reference (CTEs, subqueries, JOINs, window functions), and every function call. Not regex. The actual parser that Postgres uses to execute queries.

Agents identify themselves via application_name in the format agent:<id>:mission:<task>. The proxy checks each query against that agent's allowed tables, allowed operations, and a global blocklist of 50+ dangerous Postgres functions. Handles extended query protocol, TLS, multi-statement strings (each statement parsed individually).

I'm not going to spend more time on the architecture because the interesting part is what happened when I tried to break it.

Letting an AI try to break it

This part was inspired by Andrej Karpathy's idea of "vibe checking" code with adversarial agents — instead of writing test cases by hand, you point an LLM at your system and tell it to find holes. Except here the stakes are real: the attacker agent's job is to steal data from tables it shouldn't have access to.

I gave the attacker the database schema, told it the policy rules, and let it run. Hundreds of rounds, no human in the loop. Think of it like a chess match between two systems: FaultWall enforcing rules, the LLM trying every SQL trick it can think of to get around them.

First run, 500 rounds. The attacker figured out pretty quickly that I was only checking for table names in the obvious places — the FROM and WHERE parts of a query. But SQL lets you stick a sub-query almost anywhere. The attacker started hiding data lookups inside ORDER BY clauses, window functions, anywhere I wasn't looking.

-- Sneaking a lookup into ORDER BY
SELECT 1 FROM public.feedback
ORDER BY (SELECT password FROM public.users LIMIT 1);

674 times the attacker got data it shouldn't have. In 32 rounds. It found the blind spot and just kept exploiting it different ways.

The week of whack-a-mole

I fixed the ORDER BY gap. The attacker immediately found the next one. This pattern repeated for about a week.

It found that Postgres lets you attach filters to aggregate functions, and I wasn't checking inside those. It found that you can wrap dangerous function calls inside array literals and my parser would skip right over them — that one alone let it steal data 3,303 times before I caught it. It found that instead of calling a blocked function by name, you can cast a string to a special Postgres type called regproc and invoke the function that way, completely dodging the blocklist.

-- Hiding a blocked function call inside an array
SELECT ARRAY[(SELECT pg_read_file('/etc/passwd'))];

-- Calling a blocked function by its type instead of its name
SELECT 'pg_read_file'::regproc('/etc/passwd');

The wildest one was XMLTABLE. Postgres has this feature where you can turn XML into table rows, and inside that XML expression you can embed a query that reads from any table. My parser saw it as just a function call and didn't look inside.

Each round I learned something new about how many weird corners Postgres's SQL has. Turns out there are about 25-30 places in a SQL statement where you can hide a sub-query or function call. By the end of the week, I'd covered all of them.

Where it stands now

Latest run, 100 rounds: zero successful data thefts. The attacker still tries. It generates increasingly creative SQL. But the parser now checks every corner of the query, and the blocklist covers 50+ dangerous Postgres functions.

I won't say it's unbreakable — Postgres has a huge SQL surface area. But the problem is finite. There's only so many places in a SQL query where you can hide something, and I've now covered most of them. Each new thing the attacker finds is a smaller gap than the last.

Dashboard from a test run with 3 agents. 8 blocked violations, per-agent activity tracking, enforcement mode active.

What's still broken

Agent identity is trivially spoofable. Agents set application_name to identify themselves, and any Postgres client can set that field to anything. The proxy model helps because agents connect to FaultWall, not Postgres directly, so they never have the real DB credentials. But application_name itself has zero cryptographic guarantees. The real answer is mTLS or signed JWTs per agent. I haven't built either. I shipped without them because identity spoofing requires a compromised agent that already knows about FaultWall's naming convention, and in the confused deputy threat model the agent isn't malicious, it's just generating bad SQL from bad prompts. But if you're thinking "what about a deliberately adversarial agent?" you're right, and this is the biggest open gap.

Postgres only. The parser is Postgres's own C parser compiled into Go via pg_query_go. MySQL, SQLite, SQL Server would each need their own parser. That's a lot of work I haven't done and probably won't until there's demand.

Parser latency. Every query gets parsed before reaching the database. In my benchmarks it's under 1ms for typical queries. For agent workloads (a few queries per minute) this is noise. If you're routing high throughput application traffic through it, measure first.

Try it

Point your agents at the proxy instead of directly at Postgres:

# Run FaultWall as an inline proxy
docker run \
  -e UPSTREAM_URL=postgres://user:pass@your-db:5432/mydb \
  -e PROXY_LISTEN=0.0.0.0:5433 \
  -e POLICY_FILE=/app/policies.yaml \
  -e POLICY_ENFORCEMENT=monitor \
  -p 5433:5433 -p 8080:8080 \
  ghcr.io/shreyasxv/faultwall:main

Start in monitor mode. Watch what your agents actually do on localhost:8080. Switch to enforce when you're comfortable.
