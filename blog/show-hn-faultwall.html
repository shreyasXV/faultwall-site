Show HN: Give your AI agents a database password

An open source SQL firewall for AI agents. Per-agent, per-task policies. Single Go binary. Drops in as a Postgres proxy.

April 2026 · 4 min read

TL;DR: FaultWall is a single Go binary you put between your AI agent and Postgres. It parses every query with the real Postgres C parser and enforces a YAML policy per-agent per-task. ~0.14ms overhead on managed Postgres. MIT licensed. GitHub

I kept giving Cursor and a few other agents direct Postgres access. Most of the time it was fine. Sometimes it wasn't. One query pinned CPU for three minutes. One (in a sandbox, thankfully) tried a cross join across three big tables that would have returned a few hundred million rows.

None of that was malicious. The agent did what its prompt said. Postgres did what the agent asked. Nothing in between knew the difference between "summarize today's feedback" and "enumerate the database."

That missing layer is what FaultWall is.

What it does

FaultWall is a Go binary that sits between your agent and Postgres. Every query gets parsed through the real Postgres C parser (via pg_query_go) and checked against a YAML policy before it hits the database.

You define what each agent is allowed to do on each task:

agents:
  support-bot:
    missions:
      summarize-feedback:
        allowed_tables: [public.feedback]
        allowed_operations: [SELECT]
      update-order-status:
        allowed_tables: [public.orders]
        allowed_operations: [SELECT, UPDATE]

The agent tells FaultWall which task it is on using application_name at connect time (e.g. agent:support-bot:mission:summarize-feedback). If support-bot tries to read public.users while it is supposed to be summarizing feedback, the query is blocked and logged.

Dangerous functions, blocked by default

On top of table-level policies, FaultWall blocks about 50 Postgres functions that kept showing up in agent exfiltration paths when we red-teamed this:

  • pg_read_file, pg_read_binary_file — read arbitrary files off the server
  • lo_export, lo_import — dump DB contents to disk
  • dblink, postgres_fdw_* — exfiltrate to external databases
  • query_to_xml, schema_to_xml — serialize whole schemas (the RLS bypass story)
  • copy ... from program — shell execution

Agents can be allowlisted per function if they legitimately need one. Default is deny.

Monitor first, enforce when you are ready

Flipping straight to enforce mode in production is scary. So there are two modes:

  • Monitor — log every policy violation, don't block
  • Enforce — block and return a Postgres error to the agent

Run monitor for a week. Watch the dashboard. Find the queries that surprised you. Tune the policy. Then switch.

Monitor-mode audit log. What your agents are actually doing before you block anything.

Setup in 60 seconds

Pull the image, point it at your database, set a policy file, expose the port:

docker run \
  -e UPSTREAM_URL=postgres://user:pass@your-db:5432/mydb \
  -e PROXY_LISTEN=0.0.0.0:5433 \
  -e POLICY_PATH=/policies.yaml \
  -e POLICY_ENFORCEMENT=monitor \
  -v $(pwd)/policies.yaml:/policies.yaml \
  -p 5433:5433 -p 8080:8080 \
  ghcr.io/shreyasxv/faultwall:main

Point your agent at localhost:5433 instead of Postgres. Dashboard at localhost:8080. No SDK, no library, no app code changes.

Tested on RDS, Aurora, Neon, Supabase, Cloud SQL, pgBouncer, and self-hosted. One client-side flag (channel_binding=disable) is needed for some managed PG setups — inherent to any transparent TLS proxy, documented in the compatibility guide.

Overhead is about 0.14ms per query on managed Postgres. For LLM-bound workloads (1–10 QPS, LLM dominates latency) it is invisible.

Did we actually test this?

Yes. We pointed an autonomous AI agent (built on Karpathy's AutoResearch) at a test database and told it to steal everything. 60,000+ attack queries across 21 rounds. Every round exposed a new class of bypass — subqueries hidden in ORDER BY, UPDATE SET, XMLTABLE, COALESCE, ARRAY constructors, window FILTER clauses. Postgres's grammar has about 30 expression-bearing AST node types where this pattern applies. We walked every one.

Final versions: 0 real data exfiltrations. Full writeup with benchmarks vs RLS, pgbouncer, and ORM filtering: here.

Honest status

Single Go binary, about 3,600 lines, MIT licensed. I have been running it against my own test agents for a few weeks. The policy engine, dashboard, and managed-PG wire-level compatibility all hold up. Things not built yet: post-connect JWT identity (application_name is spoofable — fine for confused-agent threat model, not fine for adversarial insiders), a Kubernetes operator, anything other than Postgres. It is v1. Tell me where it breaks.
