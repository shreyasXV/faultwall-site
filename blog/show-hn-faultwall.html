FaultWall: Monitor and enforce what AI agents do to your database in real time

An open source firewall for AI agents on Postgres. Under 1ms latency. Compatible with every managed Postgres (RDS, Aurora, Neon, Supabase, Cloud SQL) and self-hosted.

April 2026 · 4 min read

TL;DR: Your AI agent has your database password. You probably don't know what it's running most of the time. FaultWall is a single Go binary that sits in front of Postgres, logs every query per agent, and blocks the ones that break a YAML policy. Around 0.14ms per query on managed PG. MIT licensed. GitHub.

I gave Cursor direct Postgres access for a few weeks and watched it work. Most of the queries were fine. A few were not. One pinned CPU for three minutes. Another, in a sandbox, tried a cross join across three big tables that would have returned a few hundred million rows before finishing.

Nothing about that was malicious. The prompt said what it said. The agent did what it was told. I had no idea any of it was happening until the slow query showed up in pg_stat_activity, and by then the damage (to CPU, not data, that time) was done.

FaultWall does two things. It logs every query the agent runs, grouped by which agent and which task. And it blocks anything that doesn't match a policy you wrote. That's it.

You can leave it in monitor-only mode for as long as you want. A lot of people probably will. Watching what your agents do all day is often the scary part.

Why I cared enough to build it

Postgres roles are the obvious answer and they're not good enough on their own. The same role covers every task an agent does, so a bot that summarizes feedback has the same permissions as when it updates orders. RLS helps for row-level reads but doesn't cover function-based leaks like query_to_xml, lo_export, or dblink. Those aren't table reads; RLS never looks at them.

What I wanted was simple: this agent, on this task, can do exactly these things. And I want the query log to look at in the morning.

Live dashboard. Queries grouped by agent and task. Blocked ones are red.

The policy

YAML. Per-agent, per-task. The shape is:

agents:
  support-bot:
    missions:
      summarize-feedback:
        allowed_tables: [public.feedback]
        allowed_operations: [SELECT]
      update-order-status:
        allowed_tables: [public.orders]
        allowed_operations: [SELECT, UPDATE]
        blocked_columns:
          public.orders: [internal_notes]

The agent tells FaultWall which task it's on using application_name at connect time (agent:support-bot:mission:summarize-feedback). Every query runs through the real Postgres C parser (pg_query_go) and gets checked against the policy. If it doesn't match, FaultWall kills the query and logs it with the full SQL, the agent, the task, and why it failed.

There's also a default blocklist of about 50 Postgres functions that showed up repeatedly in agent exfiltration paths during testing: pg_read_file, lo_export, dblink, query_to_xml, copy ... from program, and friends. Per-agent allowlists if you legitimately need one.

Monitor first

Flipping straight to enforce in production is scary, so monitor mode is the default. Log everything, block nothing. Run it a week, watch the dashboard, find the queries that surprised you, tune the policy. Then flip to enforce.

How it compares to RLS, pgbouncer, and ORM filtering

Before shipping I pointed an adversarial AI agent (built on Karpathy's AutoResearch harness) at a test database and told it to steal everything. 60,000 attack queries across 10 days. Same attacks run against four setups: FaultWall, Postgres RLS, ORM-layer filtering, and pgbouncer.

Methodology: 482 normalized attacks after stripping VALUES-alias spoofing from both sides. Test DB with 4 tables, 2 allowed and 2 protected (fake creds + bcrypt hashes). Agent's goal: read the protected tables. Each defense tested in isolation against the same query set. Full writeup with attack examples and raw numbers →

Defense Block Rate Real Info Leaks
FaultWall 100% 0
Postgres RLS 78.6% 103
App-layer ORM filtering 0% 482
pgbouncer 0% 482
Normalized benchmark across 482 real attack queries.

The short version is that RLS is necessary but not enough. It catches row-level reads but misses function-based exfiltration: pg_stat_activity, schema_to_xml, aclexplode, pg_cancel_backend. Those are function calls against system catalogs, not table reads, so RLS never sees them. Use RLS and FaultWall together. Full red-team writeup is here.

Honest status

Single Go binary, around 3,600 lines, MIT. I've been running it against my own agents for a few weeks. The policy engine, dashboard, and managed-PG compatibility hold up. Things I haven't built: post-connect JWT identity (application_name is spoofable, which is okay against confused agents but not against adversarial insiders), a Kubernetes operator, anything that isn't Postgres. It's v1. Tell me where it breaks.

Managed Postgres compatibility

Transparent TLS proxy, wire-level. Tested end-to-end against:

AWS RDS
AWS Aurora
Neon
Supabase
Google Cloud SQL
Self-hosted PG

Also works through pgBouncer in both transaction and session pooling modes. For some managed PG setups you'll need channel_binding=disable in your connection string. That's inherent to any transparent TLS proxy sitting between your driver and Postgres, and every SCRAM-aware driver supports the flag. It's in the compat guide.

Setup

docker run \
  -e UPSTREAM_URL=postgres://user:pass@your-db:5432/mydb \
  -e PROXY_LISTEN=0.0.0.0:5433 \
  -e POLICY_PATH=/policies.yaml \
  -e POLICY_ENFORCEMENT=monitor \
  -v $(pwd)/policies.yaml:/policies.yaml \
  -p 5433:5433 -p 8080:8080 \
  ghcr.io/shreyasxv/faultwall:main

Point your agent at localhost:5433 instead of Postgres. Dashboard at localhost:8080. No SDK or library, nothing to change in the app code. Around 0.14ms per query on managed Postgres, which for LLM-bound workloads is noise.
